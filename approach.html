Approach · how I work

I design the interface and build the system behind it.

I'm a design engineer. Designers decide what a thing should be; engineers make it real. The expensive mistakes happen in the gap between those two jobs, so that gap is where I work: close enough to design to shape what gets built, close enough to the code to ship it.

01The method

Four habits, run as one loop on every piece of work.

How I decide and pace work

Shape it

A vague ask becomes a scoped bet with a time budget. I design the solution to fit that budget, name the risks, and say what's out of scope before anyone writes code.

How I make it work for real people

Design for behaviour

A feature only matters if someone actually uses it. I find the real trigger, cut the friction between the person and the payoff, and make the behaviour easier before trying to make it more motivating. If a pattern lifts a metric by making someone feel worse, it doesn't ship.

How I know it worked

Prove it

Before building, I write down what would show it worked: an early signal like activation, and the slower outcome behind it like retention. After it ships, I go back and check. Most work never gets that second look. I make it a step.

How I make it durable

Ship it as a system

Accessible markup, motion with a reason, and components that read shared design tokens. Colour and spacing can't drift, because there is nowhere for them to drift to. This site is built that way (see the case study).

02The loop

Together they make one loop.

Shaping aims the work at the right problem. The build turns it into system components. The check at the end feeds the next round of shaping.

  1. 01
    Shape

    Scope the ask into a bet that fits a time budget, with risks and non-goals named.

  2. 02
    Design

    Start from the real trigger, remove friction, and run an ethics check on the pattern.

  3. 03
    Build

    Token-built components with accessibility handled at the system level, not per screen.

  4. 04
    Check

    Measure the signal named up front, then write down what happened for the next bet.

03What you get

From the other side of the desk.

  • Production code, not just a Figma file: modern CSS and interactions that still feel right in a real browser.
  • A design system that stays consistent, because brand and theming live in one place.
  • Accessibility and performance handled at the system level, so contrast, keyboard use and focus are right by default.
  • A half-formed idea in, a scoped and time-boxed bet out, with success defined before the work starts and checked after it ships.
04In practice

This site re-skins itself from one line of CSS.

The clearest example of the method is this site itself, because the method was used on it.

Problem & budget

Re-theming the portfolio for each company meant hand-editing CSS on every page: slow, and easy to get inconsistent. Worth one focused cycle.

The one rule

Components may only reference semantic tokens, never raw values. And no framework, no build step: plain HTML and CSS.

Build

Three stylesheets load in order: a token contract with neutral fallbacks, a swappable brand pack, and components that read only tokens. Accessibility defaults sit in the contract, so every component inherits them.

Outcome

Re-theming went from edits on every page to one line in each page's <head>. The site also ships structured data and an llms.txt index, so an AI assistant can read the system as easily as a browser can.

06Principles

A few things I hold to.

The combination is the value.

Plenty of people know one of these disciplines well. I run all four as a single way of working.

Engagement has an ethical line.

Habits are worth designing only where they help the person using the product. For tools people should open, use and close, "come back more often" is the wrong target.

The work should explain itself.

Anywhere this site makes a call, the reasoning sits a click away: the problem, the decision, and what it produced. If you leave with something you can use, the page did its job.
