# Token System — the three-layer mechanic

A page renders from three stylesheets, loaded in order: `system/tokens.contract.css` (layer 1 — every semantic token components may use, with neutral fallbacks; no brand ever) → one client pack (layer 2 — `tokens.neutral.css` by default; a company build clones it to `tokens.<company>.css`) → `system/components.css` + surface styles (layer 3 — token-only, no literals). Re-skinning a whole site = swapping the one pack `<link>` in each page's `<head>`; nothing else moves. Chrome (header/footer/nav) is injected by `system/site.js` from `window.CLIENT_CONFIG` (`client.neutral.config.js` is the neutral default). Full walkthrough: `README.md`.

**Adding a token:** add it to the `contract` group in `system/tokens.source.json` with a neutral fallback FIRST (brand values bind in the pack tiers), then regenerate: `node agent-layer/gen-token-css.mjs` — and commit the source AND the generated CSS together. A token referenced by `components.css` but missing from the contract is a bug. Brand values never enter the contract; literals never enter components.

**Machine layer:** the inversion has landed (2026-07-17, ticket #2): `system/tokens.source.json` (DTCG, string profile) is the source of truth; `agent-layer/gen-token-css.mjs` generates `tokens.contract.css` + `tokens.neutral.css` from it — never edit those two by hand (`--check` regenerates in memory and exits 1 on drift). `agent-layer/gen-tokens.mjs` remains the separate per-company projection (reads a built site's contract + pack CSS, emits DTCG `tokens.json`; a bare `var(--x)` value becomes an alias).
